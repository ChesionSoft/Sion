# Sion 会话体验、上下文统计与运行详情设计

## 背景

当前对话工作台已经有上下文圆环、会话、运行记录和持久化对话轮次，但四处体验尚未闭环：

1. 上下文圆环仅在输入框有草稿时触发估算，发送后立即清空。实际运行中还可能始终停留在“暂无估算”，现有测试没有覆盖 React、Tauri IPC 与 Rust 估算的完整链路。
2. 模型只接收最近 16 条消息，无法形成用户期望的完整会话上下文。
3. Tauri 迁移前的空会话预设和会话用量界面没有迁移完整。旧版三个预设为“梳理本节已有信息”“列出待确认问题”“基于参考资料补充细节”；旧版用量数据类型仍在 Rust 域模型中，但当前运行把消息用量写为 `None`。
4. 运行记录只能查看简短状态列表，无法检查模型、时间、上下文、活动过程、交付结果或错误详情。

本设计采用“统一会话遥测”方案：Rust 使用同一个规范化上下文构建器服务真实模型运行与界面统计，同时持久化每次模型调用的用量，并由 GPT 风格的轻量界面按需展示详情。

## 目标

- 对话主界面采用干净、聚焦的 GPT 风格，不让遥测信息干扰阅读。
- 空会话显示四个预设；点击只填入输入框，允许编辑，按回车才发送。
- 圆环持久展示当前会话已经占用的模型上下文窗口，不依赖输入框是否有草稿；真实发送会把新草稿加入后再次校验。
- 上下文弹层同时展示当前输入构成和会话累计输入、输出用量。
- 模型使用当前会话完整历史，不再静默限制为最近 16 条。
- 运行记录可点击打开居中详情弹窗。
- 统计失败不阻塞发送；发送时由 Rust 对真实输入执行权威的上下文校验。

## 非目标

- 不增加浏览器搜索、浏览器自动化、Playwright 或任何新的 Web 出站能力。
- 不实现自动摘要、上下文压缩或静默裁剪。
- 不保存完整模型 prompt 的冗余副本。
- 不为历史运行伪造当时未记录的详情。
- 不改变交付块的验证、预览、CAS 保存或原子项目存储规则。

## 用户体验

### GPT 风格对话

对话仍保持单列阅读流：用户消息使用弱背景气泡，助手消息使用无边框正文。模型、文件、上下文和发送控制集中在底部 composer；复杂信息通过弹层或弹窗按需展开。

空会话中央显示简短引导和四个预设：

1. 梳理本节已有信息
2. 列出待确认问题
3. 基于参考资料补充细节
4. 检查本节遗漏并提出改进建议

点击预设只设置 composer 草稿并聚焦输入框，不创建额外消息、不自动发送。用户可修改文本，按回车发送，Shift+Enter 换行。

### 会话上下文

composer 工具栏保留圆环入口。圆环在空输入时也显示最近一次有效会话快照：

- 低于 80%：正常状态。
- 80% 至 100%：警告状态。
- 超过 100%：阻止下一次真实发送。
- 快照读取失败：明确显示“统计暂不可用”，不显示“暂无估算”。

点击圆环打开轻量弹层，显示：

- 当前上下文：已用 token、模型窗口和百分比。
- 构成：系统及交付协议提示、节点 Agent 规则、当前节点 Markdown、完整会话历史、本轮选中文件。
- 累计用量：输入 token、输出 token、总 token、模型调用次数。
- 数据来源：精确、估算或含估算。

当前上下文占用与累计会话用量是两个独立指标。圆环表示当前持久会话及本轮所选文件占模型窗口的比例，不包含尚未提交的 composer 草稿；累计用量用于观察整个会话的模型消耗，不能用来推断窗口剩余量。发送时 Rust 会把草稿加入相同上下文并重新执行权威校验。

前端不在用户每次按键时请求统计。会话快照在以下时机刷新：

- 打开或切换会话。
- 模型选择改变。
- 所选文件改变。
- 一轮运行完成、失败或取消。
- 当前节点 Markdown 或 Agent 规则保存成功。

### 运行记录与详情

运行记录列表中的每一行均为可点击目标。对话轮次内的完成或失败状态行也可以打开同一详情。详情采用用户确认的居中弹窗，不使用右侧抽屉或列表内展开。

弹窗展示可用的以下字段：

- 运行 ID、运行类型、节点、会话和关联轮次。
- 状态与公开摘要。
- 创建、开始、结束时间和耗时。
- provider、模型和推理强度。
- 选中文件。
- 运行开始时的上下文占用和构成。
- 本轮输入、输出和总 token 以及来源。
- 活动时间线。
- 交付结果、更新章节、revision 变化或冲突。
- 失败阶段、公开错误或取消原因。

历史记录缺少字段时，该字段显示“历史记录未保存此信息”；其他详情继续可用。

## 架构

### 规范化上下文构建器

`src-tauri/src/conversation_runtime.rs` 继续承担模型输入准备，但将最近 16 条消息的窗口改为当前会话完整历史。这里的完整历史明确指持久化消息中可见的 user、assistant 和 system 文本；不发送隐藏推理内容。历史消息上的附件引用不会让文件全文在后续轮次自动重复进入 prompt，文件全文仍只在用户本轮明确选择时进入上下文。构建器接收节点、会话、模型窗口、节点规则、本轮草稿和本轮选中文件，并返回：

- `prompt`：真实发送给模型的完整输入。
- `attachments`：本轮消息需要持久化的附件引用。
- `snapshot`：与该 prompt 同源的上下文快照。

禁止在上下文查询命令和真实运行命令中分别拼装 prompt。两者必须调用同一个构建器，以保证统计、超限判断和模型实际输入一致。

新增的领域结构：

```text
ConversationContextSnapshot
  estimatedInputTokens
  contextWindowTokens
  ratio
  status
  breakdown
    protocolTokens
    rulesTokens
    nodeMarkdownTokens
    conversationTokens
    attachmentTokens
  cumulativeUsage
    inputTokens
    outputTokens
    totalTokens
    callCount
    source
  calculatedAt
```

`breakdown` 每一项使用相同的确定性 tokenizer 估算；总量由最终 prompt 整体估算，允许因分段取整产生极小差异。若需要展示可加总值，由后端提供归一化后的显示值，前端不重新计算。

### 上下文查询与发送

用会话级查询命令 `conversation_context_get` 替换当前草稿级 `agent_context_estimate`。请求包含项目、节点、会话、模型选择和本轮选中文件，但不包含 composer 草稿。返回当前持久会话和本轮所选文件已经占用的上下文快照。

真实发送仍在 Rust 内重新准备完整输入并执行最终校验。前端快照只用于展示，不是安全或正确性边界：

- 快照过期或读取失败时，发送按钮仍可用。
- Rust 在 `agent_run_start` 中使用真实草稿和文件重新构建 prompt。
- 超过模型窗口时，命令拒绝运行，不创建 run、turn 或用户消息。
- 前端保留草稿和所选文件，并展示已用、上限、超出量和主要占用来源。

这消除了当前“没有 estimate 就永远无法发送”的死锁。

### 模型用量

模型流式传输层恢复用量采集并输出统一的调用用量：

- 优先接受 provider 返回的非负、可加总且输入大于 0 的精确用量。
- provider 缺失、非法或返回输入为 0 时，按真实请求文本和最终输出做确定性估算。
- 每次调用记录类别、provider、模型、输入、输出、总量和来源。
- 会话累计用量从持久化的消息或轮次用量聚合，不依赖前端内存。

API Key、Authorization header、完整 prompt 和 provider 原始响应不得进入用量、运行详情、日志或 IPC 摘要。

### 运行详情

`AgentRun` 增加可选的 `sessionId`、`turnId`、运行开始上下文统计和本轮用量摘要，保持旧 JSON 可反序列化。详情查询命令根据 `runId` 聚合：

- durable run 元数据。
- 关联 `ConversationTurn` 的活动和交付结果。
- 关联助手消息的模型执行信息和用量。

新记录在创建 run 时持久化关联 ID 和上下文快照；运行结束时补齐用量、摘要和结束状态。更新继续使用项目目录下的原子写入，不把运行详情移出项目存储边界。

## 前端组件边界

- `ConversationPane`：组织线程、空状态、composer 和预设，不负责统计计算。
- `ConversationPresets`：渲染四个固定预设并只触发 `onSelect(text)`。
- `ContextUsageIndicator`：根据快照渲染圆环、弹层、loading、warning、blocked 和 unavailable 状态。
- `RunHistoryList`：展示可点击的运行列表，不在列表内展开详情。
- `RunDetailDialog`：根据详情数据展示元信息、上下文、用量、活动时间线和交付结果。
- `App`：按会话作用域加载和刷新快照、运行详情，继续使用 scope 防止异步响应写入错误项目、节点或会话。

组件只通过 `src/api.ts` 调用 Tauri 命令；React 不直接读取项目文件或访问模型网络。

## 数据流

### 打开会话

1. 加载会话、消息、轮次和运行列表。
2. 根据会话保存的模型选择请求上下文快照。
3. 圆环显示当前完整会话占用和累计用量。

### 选择预设

1. 用户点击预设。
2. 预设文本写入 composer 并获得焦点。
3. 不请求逐键统计，不创建消息。
4. 用户编辑后按回车提交。

### 发送消息

1. 前端提交草稿和本轮选中文件。
2. Rust 读取完整持久会话，构建真实 prompt 和快照。
3. 若超限，返回结构化错误且不写入任何半成品。
4. 若可发送，原子创建 run、turn 和用户消息，再启动模型流。
5. 模型结束后持久化助手消息、用量、活动和交付结果。
6. 前端收到结束事件后刷新上下文快照和运行记录。

### 查看运行详情

1. 用户点击运行行或轮次状态行。
2. 前端按 `runId` 请求详情并打开居中弹窗。
3. 加载中显示弹窗骨架；失败时保留弹窗并显示可重试错误。
4. 关闭后焦点返回触发行。

## 错误处理

- 上下文快照失败：显示“统计暂不可用”，不阻止发送。
- 真实输入达到 80%：显示警告但允许发送。
- 真实输入超过 100%：Rust 拒绝发送；保留草稿与文件。
- provider 用量无效：回退估算并标记来源，不丢弃回复。
- 运行详情聚合失败：弹窗显示错误和重试入口，不影响对话。
- 旧运行缺字段：局部显示不可用，不让整个详情失败。
- 会话、节点或项目切换：丢弃旧 scope 的快照和详情响应。
- 所有公开错误不得包含 API Key、Authorization header 或完整 prompt。

## 兼容与迁移

- 新增字段使用 serde 默认值或 `Option`，旧会话、消息和运行记录继续读取。
- 不批量重写旧项目。旧数据在下一次正常写入时按现有存储策略更新。
- 旧运行详情只展示已存在字段。
- 保持项目数据只写入 `<projects directory>/<project id>/`，全局 provider 配置仍只在 `~/.sion/`。

## 测试

### Rust 域与运行时

- 完整会话历史全部进入 prompt，超过 16 条时不再截断。
- prompt 与快照来自同一次构建，最终估算等于真实 prompt 估算。
- breakdown 覆盖协议、规则、节点、会话和附件。
- 80% 和 100% 阈值行为正确。
- provider 精确用量校验、输入为 0 回退和估算来源正确。
- 累计用量跨多轮聚合正确。

### 存储与 Tauri 命令

- 新运行关联 session、turn、上下文和用量，并可在重启后恢复。
- 旧 run JSON 缺少新字段仍可读取。
- 上下文查询与真实发送共享构建器。
- 超限发送不会创建 run、turn 或用户消息。
- 运行详情可以聚合完整记录，也能返回兼容的部分旧记录。

### React

- 四个预设均可见，点击只填入文本，不自动发送。
- 空草稿仍显示会话快照。
- 不随每次键盘输入请求快照。
- 会话、模型、文件、运行完成和节点保存会触发正确刷新。
- unavailable 状态不禁用发送。
- 运行列表行和轮次状态行打开同一居中弹窗。
- 弹窗支持加载、失败、重试、关闭和焦点恢复。

## 验收标准

1. 新空会话显示四个确认过的预设，点击后文本进入输入框，回车才提交。
2. 没有草稿时圆环仍展示当前会话上下文，且不再出现无解释的“暂无估算”。
3. 弹层同时展示当前窗口占用构成与累计输入、输出用量，并标明精确或估算来源。
4. 第 17 条及之后的历史消息仍进入下一轮真实模型输入。
5. UI 快照失败不会锁死发送；真实超限由 Rust 阻止且不产生半条记录。
6. 每条运行记录可打开居中详情弹窗；新记录显示完整详情，旧记录优雅降级。
7. API Key、完整 prompt 和 provider 原始响应不出现在项目数据、运行详情、日志或 IPC 摘要中。

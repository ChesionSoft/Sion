# 对话模型、推理、上下文与文件引用设计

**日期：** 2026-07-17
**状态：** 对话设计已批准，等待书面规格审阅

## 背景

Sion 的 Rust Provider 数据结构已经允许一个供应商保存多个模型，但当前 React 编辑器只提交一个默认模型，保存时会覆盖原模型列表。当前 Agent Run 也始终解析全局默认模型，对话页没有会话级模型和推理强度选择。项目文件池已经能导入、提取和勾选文件，Agent Run 也接受 `fileIds`，但新版对话输入区没有直接选择、导入或展示本次文件引用的入口。

本设计把这三个缺口合并为一条完整的对话运行参数链：供应商维护多个模型；会话保存模型和推理强度；单条用户消息临时附加文件；Rust 在入队时校验并冻结模型、推理和文件；对话历史展示实际使用信息。每个模型还必须声明上下文窗口，输入框通过紧凑圆环提示预计输入占用。

本规格取代 `2026-06-14-sion-model-files-agent-config-design.md` 中与模型列表、对话模型选择和文件引用有关但尚未落地或已被新版界面覆盖的部分。其他已有功能不受影响。

## 目标

- 一个 Provider 可手动维护多个模型，且恰好有一个默认模型。
- 每个模型记录上下文窗口；未配置上下文窗口的旧模型不能发送请求。
- 对话输入框按已确认的两级菜单选择模型和推理强度。
- 模型和推理选择按会话持久化，新会话继承全局默认模型。
- 推理强度统一为 `关闭 / 低 / 中 / 高`，不维护模型能力开关。
- 输入框可从项目文件池多选文件，也可导入新文件；文件只用于下一条消息。
- 用户消息保留文件名快照，Assistant 回复保留实际模型和推理强度。
- 运行入队时冻结模型、推理强度和文件，后续配置修改不改变已入队任务。
- Rust 使用正式 Prompt 组装逻辑估算输入上下文；80% 预警，超过窗口时阻止发送。
- 保持 API Key、本地文件正文、模型请求体和模型网络访问在现有 Rust/Tauri 边界内。

## 非目标

- 不从供应商 API 自动发现模型。
- 不创建模型、文件或对话配置预设。
- 不限制最大输出 token，也不向供应商发送 `max_tokens`、`max_output_tokens` 或同类输出上限。
- 不自动截断旧消息、节点内容、Agent 规则或附件以适配上下文窗口。
- 不为不同模型维护自定义推理档位或“是否支持推理”开关。
- 不增加浏览器搜索、浏览器自动化、Playwright 或桌面运行时 Web 浏览能力。
- 不改变文件池现有提取格式与受限预览策略。

## 已选方案

采用端到端持久化扩展。Provider、ChatSession、ChatMessage 和 AgentRun 都记录各自职责所需的非秘密元数据，Rust/Tauri 继续作为唯一权威边界。

未采用的方案：

1. **仅前端保存选择。** 改动较小，但重启和切换会话后状态会丢失，也不能可靠追溯旧回复。
2. **引入对话配置预设。** 扩展性较强，但需要预设生命周期和失效处理，超出本次需求。

## 架构与职责

### React

React 负责展示和收集以下内容：

- Provider 编辑器中的多模型列表和上下文窗口；
- 当前会话的模型与推理强度；
- 下一条消息的临时文件选择；
- Rust 返回的上下文估算状态；
- 历史消息中的文件标签和实际运行元数据。

React 不读取文件正文，不解析 API Key，不自行构造模型网络请求，也不把供应商密钥写入项目状态。

### Tauri 命令层

Tauri 命令层负责：

- 校验并保存 Provider 和多个模型；
- 原子更新会话的模型选择；
- 根据当前草稿和文件选择计算预计输入；
- 在发送入口重新执行权威校验；
- 解析本地 Provider 密钥和文件文本；
- 保存带元数据的消息与运行记录；
- 将统一推理强度映射到具体协议请求。

### Rust Domain 与 Storage

`sion-core` 定义向后兼容的数据类型、推理强度枚举、文件引用快照和运行元数据。`sion-storage` 继续使用现有项目目录、原子写入和直接 UUID 路径规则保存会话、消息、文件和运行。Provider 配置仍只写入 `~/.sion/providers.json`，API Key 使用现有受限权限和不回显边界。

### Agent Transport

`sion-agent` 接收已经解析并冻结的 endpoint、协议、模型、推理强度和 Prompt。Transport 不重新读取全局默认模型，也不在运行开始后重新解释会话选择。

## 数据模型

### ProviderModel

Provider 模型记录包含：

- `name: string`
- `isDefault: boolean`
- `toolCalling: boolean`，保留现有字段和行为
- `contextWindowTokens: number | null`

新建或保存 Provider 时，每个模型名称必须非空且在该 Provider 内唯一，恰好一个模型为默认模型。新模型必须填写正整数上下文窗口。旧记录缺少该字段时反序列化为 `null`，仍可列出和编辑，但不能被对话选择或用于发送。

Provider 编辑仍允许 API Key 留空以保留现有密钥。模型列表修改不得造成密钥回显或把密钥带入普通 IPC 列表结果。

### ReasoningEffort

统一枚举为：

- `off`
- `low`
- `medium`
- `high`

所有已完成上下文配置的模型都显示四个选项，不维护 `supportsReasoning`。新会话默认 `medium`。切换模型时保留当前会话的推理强度；用户可以显式切换为 `off`。

### ChatSession

会话新增可选模型选择：

```text
modelSelection:
  providerId
  model
  reasoningEffort
```

新会话从全局默认 Provider 的默认模型初始化，并使用 `medium`。旧会话缺少该字段时，界面临时解析当前全局默认值；第一次修改模型、修改推理强度或发送时将选择原子写入会话。

若会话引用的 Provider 或模型后来被删除，Sion 不静默切换到其他模型。输入框显示选择失效状态并要求用户重新选择。

### ChatMessage

用户消息新增可选附件快照列表。每项只保存：

- `fileId`
- `originalName`

保存原始文件名是为了在文件后来从文件池删除或不可用时仍能展示历史标签。消息不保存附件正文、绝对路径或额外文件副本。

Assistant 消息新增可选执行元数据：

- `providerId`
- `model`
- `reasoningEffort`

这组字段记录回复实际使用的值，不随会话以后切换模型而改变。历史消息缺失这些字段时按旧样式渲染。

### AgentRun

运行记录新增：

- `providerId`
- `model`
- `reasoningEffort`
- `fileIds`

这些字段在入队时冻结。运行记录不保存 API Key、endpoint、文件正文、完整 Prompt 或完整请求体。

## Provider 多模型编辑器

现有 Provider 表单保留名称、API URL、URL 模式、协议和 API Key。单个“默认模型”文本框替换为模型列表编辑区。

每个模型行包含：

- 模型名称输入框；
- 上下文窗口 token 输入框；
- 默认模型单选；
- 删除动作。

编辑区提供“添加模型”。模型可手动添加、重命名和删除，不调用供应商发现接口。保存前校验：

- 至少一个模型；
- 名称去除首尾空白后非空；
- 同一 Provider 内名称唯一；
- 每个新模型都有正整数上下文窗口；
- 恰好一个默认模型。

旧模型缺少上下文窗口时显示“待补充限制”。它仍可保存在列表中，但 Provider 保存操作要求用户补齐所有现存模型的上下文窗口，避免继续制造不完整配置。

## 对话模型与推理选择

输入框右下角显示一个组合按钮，摘要为“模型名 · 推理强度”。交互按参考图分两级：

1. 首次点击只打开左侧主菜单，包含“模型”和“推理强度”。
2. 点击主菜单某一行，才在右侧展开对应子菜单。

模型子菜单列出所有 Provider 下已配置上下文窗口的模型。重名模型以 Provider 名称辅助区分。缺少上下文窗口的模型不允许选择，并显示“请先在设置中补充上下文窗口”。

推理子菜单始终显示 `关闭 / 低 / 中 / 高`：

- `off`：请求不包含推理参数；
- `low / medium / high`：协议适配器发送相应参数；
- 供应商或模型不支持时，运行失败信息明确建议切换为“关闭”后重试。

模型和推理强度按 ChatSession 保存，而不是按节点或单条消息保存。新会话继承全局默认模型并使用 `medium`，之后记住该会话自己的选择。

## 单条消息文件引用

输入框左下角“＋”打开文件菜单。菜单顶部提供“导入新文件到文件池”，其后列出当前项目文件池。

- 文本可用文件可多选。
- 提取失败或格式不支持的文件禁用，并显示原因。
- 导入的新文件若文本可用，则自动加入本次选择；否则只进入文件池并显示不可附加原因。
- 已选文件以紧凑标签显示在输入框中，可逐个移除。
- 文件只用于下一条消息，不成为会话持续上下文。

权威校验失败时保留草稿和全部文件标签。消息成功持久化后清空标签；即使随后网络调用失败，文件仍通过用户消息的附件快照在历史中可见。

## 上下文窗口与紧凑提示

### 估算范围

Rust 使用与正式 Agent Run 相同的 Prompt 组装函数生成待估算文本，包含：

- 系统指令；
- 当前节点 Agent 规则和项目覆盖规则；
- 当前节点标识与 Markdown；
- 当前实现纳入 Prompt 的最近会话消息；
- 当前用户草稿；
- 本次选中文件的提取文本。

文件正文只在 Rust 内参与计算。React 只收到预计 token 数、模型上下文窗口、占用比例和状态。

在没有供应商专用 tokenizer 的情况下，使用确定性的保守估算：ASCII 字节按每 4 字节一个 token 向上取整，其他 Unicode 字符按每字符一个 token，合计后增加 15% 安全余量并向上取整。界面始终标注为“预计”，供应商仍是实际上下文限制的最终裁定者。

### 阈值

- 低于 80%：中性或绿色圆环，允许发送。
- 80% 至 100%：橙色圆环，允许发送并显示接近上限提示。
- 超过 100%：红色圆环，禁用发送。

本地比较仅针对预计输入，不预留或限制输出。模型请求不携带最大输出参数。供应商若因为实际 token 计算或输入输出组合限制返回上下文错误，Sion 将其转换为可读失败提示。

### 界面

输入框不显示长进度条。模型选择按钮旁显示一个小圆环图标，圆环表达占用比例；悬停、聚焦或点击后显示“预计输入 / 上下文窗口”及组成说明。超限提示建议移除文件、缩短会话内容或选择更大上下文模型。

## 发送数据流

1. 用户切换模型或推理强度时，React 调用原生命令原子更新当前 ChatSession。
2. 草稿、模型、会话或文件选择变化后，React 防抖调用上下文估算命令。
3. Rust 读取会话选择、节点、Agent 规则、最近消息和文件文本，返回估算结果。
4. 用户发送时，原生编排入口重新计算并执行权威校验，不信任前端缓存的比例。
5. 校验通过后保存带附件快照的用户消息，冻结模型、推理和文件 ID，创建并持久化 AgentRun。
6. 入队任务携带进程内解析的 Provider endpoint 与 API Key，但运行记录不保存它们。
7. 运行开始后，Transport 按 Provider 协议构造 SSE 请求。
8. 完成时保存带实际执行元数据的 Assistant 消息；失败或取消时保存对应运行状态。

权威发送校验包括：

- Provider 和模型仍然存在；
- 模型已配置上下文窗口；
- 会话模型选择已持久化；
- 文件存在且提取文本可用；
- 预计输入不超过上下文窗口；
- 当前项目节点没有正在运行或排队的修改任务。

## 协议映射

`ReasoningEffort` 在 Transport 边界统一处理：

- `off` 不写推理字段；
- Chat Completions 协议使用其推理强度字段；
- OpenAI Responses 协议使用其推理对象；
- 两种协议均不写最大输出字段。

协议适配器只负责请求形状，不判断模型是否支持推理。供应商拒绝推理字段时，Sion 保留失败运行并提示用户将推理改为“关闭”。

## 错误处理

### 配置错误

- Provider 没有模型、模型名重复、缺少默认模型或上下文窗口无效时拒绝保存并在字段旁显示错误。
- 旧模型未补充上下文窗口时不可选择或发送。
- 当前会话引用已删除的 Provider 或模型时不自动回退，要求重新选择。

### 文件错误

- 提取失败和不支持的文件在选择菜单中禁用。
- 已选文件在发送前被删除或变为不可读时，权威校验失败并保留草稿和其他标签。
- 导入取消不改变文件池或当前选择。

### 上下文错误

- 估算命令失败时禁止发送并提供重试，防止绕过已确认的硬限制。
- 预计输入超过窗口时禁止发送并保留全部编辑状态。
- 供应商返回实际上下文错误时，用户消息保留，运行标记失败，并给出减小输入或更换模型的建议。

### 运行错误

- 配置、文件、上下文或节点忙校验失败时，不保存消息、不创建运行。
- 校验通过且用户消息已保存后发生网络或供应商失败时，消息和附件快照保留，运行记录标记失败。
- 推理字段不受支持时，提示改为“关闭”重试，不自动修改会话选择。
- 已入队运行使用冻结参数；修改 Provider、会话或文件选择不改变它。

## 组件边界

| 单元 | 职责 |
|---|---|
| `ProviderEditorDialog` | Provider 基础字段、多模型增删改、默认模型和上下文窗口校验。 |
| `ConversationPane` | 消息列表、文件标签、紧凑上下文圆环和发送/停止控制。 |
| `ConversationModelMenu` | 已确认的两级模型与推理菜单、会话选择更新和失效状态。 |
| `ConversationFileMenu` | 文件池多选、导入新文件、禁用原因和单次选择管理。 |
| `ContextUsageIndicator` | 圆环状态、键盘可访问的详情浮层和超限提示。 |
| `App.tsx` | 协调原生命令和页面状态，不读取文件或访问模型网络。 |
| Tauri conversation commands | 会话选择保存、上下文估算、发送权威校验与运行创建。 |
| `sion-core` | 向后兼容的数据类型、推理枚举和 Prompt/估算纯逻辑。 |
| `sion-storage` | 原子保存 Provider 之外的项目会话、消息和运行数据。 |
| `sion-agent` | 冻结后的请求参数、协议映射、SSE、取消和失败状态。 |

## 兼容与迁移

- Provider 文件继续位于 `~/.sion/providers.json`。Schema 版本升级后，旧模型缺失的上下文窗口读为 `null`，不会猜测 128K 或其他默认值。
- 旧 ChatSession 缺少模型选择时仍可读取；首次修改或发送时写入新字段。
- 旧 ChatMessage 缺少附件和执行元数据时按空值处理。
- 旧 AgentRun 缺少冻结参数时仍可展示原状态，不尝试重建历史模型信息。
- 不批量重写现有项目。所有升级在用户正常保存对应记录时发生。
- API Key 继续只在 Rust 进程内解析，永不进入项目数据、导出、日志或普通 IPC 摘要。

## 测试计划

### TypeScript 与界面逻辑

- 多模型表单：新增、重命名、删除、名称去重、唯一默认模型和上下文窗口校验。
- 两级菜单：首次只开主菜单，子菜单按需向右展开，键盘与焦点返回正确。
- 会话状态：模型与推理选择切换、恢复和失效模型提示。
- 文件选择：多选、导入后自动选择、移除、校验失败保留、消息保存后清空。
- 上下文圆环：低于 80%、等于 80%、等于 100% 和超过 100% 的状态映射。
- 历史渲染：用户文件标签和 Assistant 实际模型/推理元数据。

### Rust Domain 与 Storage

- Provider 多模型校验、默认归一化、上下文窗口持久化和旧 Schema 读取。
- ChatSession 模型选择的新增、更新、原子保存和旧数据读取。
- ChatMessage 附件快照、Assistant 执行元数据和旧消息读取。
- AgentRun 冻结参数的持久化和旧运行读取。
- Prompt 组装与估算使用相同输入；ASCII、中文、附件和 15% 安全余量计算确定。
- 79.99%、80%、100% 和超过 100% 的阈值行为。

### Native 编排与 Transport

- 配置缺失、模型删除、上下文未配置、文件不可读、超限和节点忙均不保存消息或运行。
- 校验通过后消息、附件、冻结参数和运行一致保存。
- 入队后改变会话或 Provider 不影响运行参数。
- `off / low / medium / high` 在 Chat Completions 和 OpenAI Responses 下生成正确请求形状。
- 所有请求都不包含最大输出限制。
- 推理不受支持、上下文拒绝、普通 HTTP 失败、SSE 失败和取消都生成正确运行状态。

## 验收标准

- 一个 Provider 能保存并重新打开多个模型，默认模型与上下文窗口保持准确。
- 缺少上下文窗口的旧模型明确不可发送，不使用猜测值。

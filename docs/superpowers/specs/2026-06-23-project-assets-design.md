# Sion 项目资产二期设计

## 1. 背景

一期已经完成对话体验升级：Agent 状态、Markdown 回复、交付稿预览、token 用量和桌面布局都已有基础闭环。用户接下来需要把“项目文件池”和“历史会话”做成可管理的项目资产能力。

当前文件池后端已经能保存上传文件，并能读取 `.txt`、`.md`、`.json`、`.csv`、`.log` 等纯文本文件，但前端上传入口只暴露 Markdown。PDF、Word、Excel 会被保存为不支持状态，无法被 Agent 引用。会话删除 API 已存在，前端缺少删除入口和删除后的会话切换策略。会话创建时目前最多保留 10 条，并会自动裁剪旧会话，用户无法主动管理不需要的会话。

## 2. 目标

- 支持上传和管理 PDF、DOCX、Excel、Markdown、纯文本、JSON、CSV、日志等项目资料。
- 对可解析文件在服务端抽取文本，让 Agent 可以在聊天中引用。
- 文件池明确展示文件类型、解析状态、大小、字数和格式相关摘要。
- 解析失败或暂不支持的文件仍保留在文件池，但不能被聊天勾选引用。
- 用户可以删除不想保留的历史会话。
- 删除当前会话后，界面自动切到同节点最近会话；没有会话时进入清晰空状态。
- 将每个节点的会话保留上限从 10 条提高到 50 条，并让规则在界面上可见。
- 沿用本地优先架构，不引入数据库、云存储或外部解析服务。

## 3. 非目标

- 不做完整文档预览器，不渲染 PDF 页面、Word 版式或 Excel 表格 UI。
- 不做 OCR。扫描版 PDF 如果没有可提取文本，应显示解析失败。
- 不支持 PowerPoint、图片、压缩包或音视频解析。
- 不做跨节点会话搜索、批量归档或会话重命名。
- 不把文件内容同步到外部服务。
- 不在本期实现后台异步任务队列；解析随上传请求同步完成。

## 4. 设计方向

采用“本地解析闭环”方案。上传时服务端保存原始文件，然后根据扩展名和 MIME 类型选择文本抽取器。抽取成功后保存一份纯文本副本，聊天引用继续读取这份文本副本。抽取失败不回滚原始文件，而是保存失败状态和可读原因。

项目文件池从“Markdown 附件列表”升级为“项目资料库”。它仍不是完整网盘，也不是文档预览系统；它的核心价值是让 Agent 稳定获得用户提供的项目资料。

会话管理从“自动保留少量最近会话”升级为“用户可主动清理”。后端已经具备删除能力，二期重点补齐前端入口、确认流程和删除后的状态恢复。

## 5. 文件资产模型

### 5.1 支持格式

| 类别 | 扩展名 | 行为 |
| --- | --- | --- |
| Markdown | `.md`, `.markdown` | 直接按 UTF-8 文本读取 |
| 纯文本 | `.txt`, `.log` | 直接按 UTF-8 文本读取 |
| 结构化文本 | `.json`, `.csv`, `.tsv` | 直接读取，必要时加简短标题说明 |
| PDF | `.pdf` | 抽取页面文本，记录页数 |
| Word | `.docx` | 抽取段落和表格文本 |
| Excel | `.xlsx`, `.xls` | 抽取工作表文本，记录工作表数 |
| 暂不支持 | 其他扩展名 | 保存原始文件，标记暂不支持 |

`.doc` 属于旧二进制 Word 格式，Node 侧稳定解析成本较高。本期允许上传但标记“暂不支持”，避免给用户造成可引用错觉。后续如确有需求，再单独评估转换链路。

### 5.2 ProjectFile 字段

现有 `ProjectFile` 继续兼容。新增字段应全部可选，以便旧项目无需迁移即可打开：

- `kind`: `markdown`、`text`、`json`、`csv`、`pdf`、`word`、`excel`、`unsupported`。
- `extractionStatus`: `available`、`failed`、`unsupported`。
- `textPath`: 成功抽取后的纯文本副本路径。沿用旧字段名称，避免重写聊天读取链路。
- `extractionError`: 面向用户的短错误原因。
- `pageCount`: PDF 页数。
- `sheetCount`: Excel 工作表数量。
- `characterCount`: 抽取文本字符数。

旧记录只有 `status` 和 `textPath` 时，按现有规则展示：`status === "available"` 且存在 `textPath` 的文件可引用。

### 5.3 状态语义

文件池展示三种用户可理解状态：

- `可引用`: 已成功抽取文本，聊天附件选择器允许勾选。
- `解析失败`: 原始文件已保存，但无法抽取可靠文本，聊天附件选择器禁用。
- `暂不支持`: 文件格式本期不解析，聊天附件选择器禁用。

后端仍以服务端记录为事实来源。聊天 API 必须重新校验文件是否可引用，不能相信前端提交。

## 6. 文本抽取

### 6.1 抽取接口

在 `src/lib/project` 内新增统一抽取边界 `extractFileText(input)`。调用者提供文件名、MIME 类型和 Buffer，得到结构化结果：

- 成功：文本、文件类别、字符数、页数或工作表数等格式摘要。
- 失败：文件类别、失败状态和错误原因。
- 不支持：文件类别和不支持原因。

`FileStore.uploadFile` 负责保存原始文件、调用抽取器、写入文本副本和更新 index。

### 6.2 格式处理

Markdown、纯文本、JSON、CSV、TSV、日志按 UTF-8 读取。对于 CSV/TSV 不做复杂表格建模，只保留原始文本，避免破坏用户数据。

PDF 使用本地 Node 可用的解析库提取文本。若 PDF 只有图片或提取文本为空，标记解析失败，原因显示“PDF 未包含可提取文本”。

DOCX 使用本地解析库提取段落和表格文本。空文档或损坏文件标记解析失败。

XLSX/XLS 使用本地解析库遍历工作表，将每个工作表转成紧凑文本块。每个工作表包含标题行，格式为 `# Sheet: <工作表名>`，再输出非空单元格内容。空工作簿标记解析失败。

### 6.3 大小与内容限制

为保护本地内存和模型上下文，本期设置保守限制：

- 单文件上传上限：20 MB。
- 单文件抽取文本上限：200,000 字符，超出后截断并记录 `truncated: true`。
- 聊天引用多个文件时，继续由服务端拼接上下文；若未来需要更细的上下文预算，可在后续阶段增加按文件裁剪策略。

## 7. 文件池 UI

上传入口文案改为“上传项目资料”，接受二期支持的扩展名。说明文案明确“可解析的文件会被 Agent 在聊天中引用”。

文件行展示：

- 文件名和类型 badge；
- 状态 badge；
- 文件大小；
- 字符数；
- PDF 页数或 Excel 工作表数；
- 上传时间；
- 删除按钮。

聊天附件选择器只展示或只允许勾选 `可引用` 文件。解析失败和暂不支持文件可以在文件池中管理，但不进入聊天上下文。

删除文件沿用现有删除能力。删除时移除原始文件和抽取文本副本；如果抽取文本副本已经丢失，删除仍应成功。

## 8. 会话管理

### 8.1 保留策略

每个项目节点最多保留 50 条会话。创建新会话时，如果超过上限，仍可裁剪最旧会话，但前端必须展示“最多保留 50 条，可手动删除”的规则。二期不做归档和恢复。

### 8.2 删除入口

会话下拉菜单中的每条会话提供删除入口。删除属于破坏性操作，必须有确认流程，确认文案包含会话名称或时间，避免误删。

删除非当前会话后，只刷新会话列表，不影响当前消息。

删除当前会话后：

1. 前端调用 DELETE API。
2. 重新获取当前节点会话列表。
3. 如果还有会话，切换到最近更新或最近创建的会话并加载消息。
4. 如果没有会话，清空消息区，`activeSessionId` 置空，显示空状态。
5. 用户发送下一条消息或点击新会话时，按现有流程创建新会话。

删除失败时保留当前 UI 状态，并展示错误。

## 9. 数据流

### 9.1 文件上传

1. 用户选择项目资料文件。
2. 前端提交 multipart form-data。
3. 路由校验项目存在和文件大小。
4. `FileStore` 保存原始文件。
5. 抽取器根据格式抽取文本或返回失败状态。
6. 成功时写入文本副本，并把记录标记为可引用。
7. 失败或不支持时保存原始文件和状态原因。
8. 前端刷新文件池列表。

### 9.2 聊天引用

1. 用户在附件选择器勾选可引用文件。
2. 前端提交 file IDs。
3. 聊天 route 对每个 ID 调用 `FileStore.readFileContent`。
4. 后端忽略不可引用、丢失或读取失败文件，并可发送 `reading_files` activity。
5. 成功读取的文本拼入模型上下文。

### 9.3 会话删除

1. 用户在会话菜单点击删除。
2. 前端展示确认。
3. 确认后调用 DELETE session API。
4. 前端刷新同节点 session index。
5. 按是否删除当前会话决定保持、切换或进入空状态。

## 10. 错误与兼容

- 损坏文件、加密文件、空文件、扫描版 PDF 和无法识别编码都不能导致上传请求整体失败。
- 路由级错误只用于项目不存在、没有选择文件、文件超出大小限制等请求级问题。
- 抽取错误原因应短而具体，不暴露本地绝对路径或内部堆栈。
- 旧文件 index 不要求迁移；缺少新字段时按旧字段推断。
- 旧会话继续可读；删除 API 已存在，二期只补齐用户入口和状态处理。
- 删除当前会话时如果刷新列表失败，保留原 active session 提示删除状态不确定，避免直接清空用户界面。
- 文件名只用于展示，不参与路径拼接；存储文件名继续使用 UUID。

## 11. 测试策略

行为变更遵循测试先行。测试至少覆盖：

- 文件类型识别：Markdown、文本、JSON、CSV、PDF、DOCX、XLSX、旧 `.doc` 和未知格式。
- 文本文件上传后保持旧 `textPath` 读取兼容。
- PDF 有文本时可抽取，扫描或空文本时标记解析失败。
- DOCX 段落和表格文本可抽取。
- XLSX 多工作表文本可抽取，并记录 `sheetCount`。
- 损坏 PDF、DOCX、XLSX 上传成功但状态为解析失败。
- 删除文件时同时清理抽取文本副本。
- 聊天 route 只读取可引用文件，不把失败或不支持文件放入模型上下文。
- 文件池 UI 展示类型、状态、大小、字符数和格式摘要。
- 聊天附件选择器禁用或隐藏不可引用文件。
- 会话上限从 10 调整为 50。
- 会话菜单删除非当前会话后不清空当前消息。
- 删除当前会话后自动切到最近会话。
- 删除最后一条会话后进入空状态，并能继续新建或发送消息。
- DELETE session 失败时 UI 保持稳定并显示错误。

完成实现后运行：

```bash
npm run test
npm run lint
npm run typecheck
```

## 12. 验收标准

- 用户可以上传 PDF、DOCX、XLSX、XLS、Markdown 和常见文本资料。
- 可抽取的 PDF、DOCX、XLSX 能被聊天勾选并提供给 Agent 阅读。
- 解析失败和暂不支持的文件在文件池中清晰可见，但不会进入模型上下文。
- 文件池能展示文件类型、状态、大小、字符数和 PDF/Excel 摘要。
- 用户可以删除不想要的历史会话。
- 删除当前会话后界面不会停留在不存在的 session。
- 每个节点最多保留 50 条会话，规则对用户可见。
- 旧项目、旧 Markdown 文件和旧会话无需迁移即可继续使用。

## 13. 后续阶段

后续可以独立设计：

- 文件内容预览、搜索和按片段引用。
- OCR 和图片类资料解析。
- PowerPoint 支持。
- 会话重命名、收藏、归档和跨节点搜索。
- 按模型上下文长度做文件内容预算和自动裁剪。
